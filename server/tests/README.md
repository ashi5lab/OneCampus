# Tests

**These are integration tests, not unit tests.** They make real HTTP requests against a running `node server.js` (or `npm run dev`), which talks to the real Railway dev database and the three provisioned dev tenants (`dev.onecampus.local`, `dev2.onecampus.local`, `dev3-kg.onecampus.local`) and their seeded test accounts documented in `HANDOFF.md` §2. There's no mocking, no test database, no fixtures beyond what already exists in those tenants.

**To run:** start the server (`npm run dev` in one terminal), then `npm test` in another. `npm test` does **not** start the server itself.

**Why this shape, not a hermetic unit-test suite:** the things most worth protecting against regression right now are cross-cutting integration concerns — tenant isolation via the per-request pinned `req.db` client, JWT tenant-scoping, the permission matrix, module-toggle gating — none of which are meaningfully testable by mocking `req.db`, since the bug class they guard against (two requests landing on different physical connections, a token replayed against the wrong tenant) only manifests against a real connection pool and real tenant schemas. Setting up a fully isolated ephemeral test database was out of scope for this pass; see `HANDOFF.md` for that as a follow-up.

**Known tradeoffs:**
- Tests run with `--runInBand` (serially, not in parallel) because several tests read-then-write shared state in a real tenant (e.g. counting `/units` before and after a create) — parallel workers would race each other.
- `tenantIsolation.test.js`'s isolation test creates a throwaway unit in `dev2.onecampus.local` on every run and never cleans it up — acceptable for a dev-only database, not something to point at a shared/production instance.
- **The suite hits the real login rate limiter (10 attempts/15min per tenant+IP, see `server/middleware/rateLimiters.js`) — this was discovered the hard way while writing these tests, not designed in from the start.** `tests/helpers.js`'s `getToken(tenant, username, password)` caches successful logins in a `Map`, so use it (not raw `login()`) everywhere a test just needs a valid token, not to assert on login behavior itself. The cache is **per test file**, not global — Jest gives each test file its own module registry even under `--runInBand`, so a `getToken` call for the same identity in two different files still logs in twice. As currently written this keeps a full suite run to ~8 real `POST /auth/login` calls, under the limit — but it has no safety margin for re-running the suite twice in the same 15-minute window, or for manual curl testing against the same tenants in parallel. If you add more test files that log in as `test_admin`, budget for it.
- **No rate-limiting test.** An automated test that intentionally triggers the 15-minute lockout would make itself (and anyone re-running the suite within that window) fail on the next run. This was verified manually once (see PR #8) but isn't covered here. A proper test would need a disposable tenant provisioned per run.
- Coverage is deliberately narrow: tenant isolation, cross-tenant JWT rejection, the permission matrix, and module-toggle gating — the four things flagged in `HANDOFF.md` as most likely to silently regress. It does not cover every endpoint, every validation rule, or the frontend.
